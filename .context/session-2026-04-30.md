## What Started This Session

---

## Phase 1–5 Recap: The Plan

### User Requirements (crystallized during grilling)
1. One centrally-styled and defined "table" component — no duplicated code, no UI inconsistency risk.
2. Foresee adding sorting, filtering, pagination — most likely server-side.
3. Columns and data represented as **data structures detached from the table component** (ProTable / TanStack mental model).
4. "Ghost" is a non-default variant of the table itself.
5. Server-rendered fallback desired but can come later.
6. User does not know frontend testing well — asked me to lead.

### Option Selected: **Option A — React DataTable Island**
- Go templates define column config as arrays of dicts.
- Raw domain data (structs) JSON-serialized into `<script type="application/json">` blocks.
- React hydrates each `[data-table-root]` independently.
- Variant system: `default` (card wrapper, gray thead) vs `ghost` (no wrapper, `#fdfdfd` thead).
- Action column supports `link`, `form`, and `button` types with `{id}`/`{name}` param replacement.
- Icon-only buttons supported via inline SVGs (`pencil`, `trash`).

## Files Created

### 1. `ui/src/components/DataTable.tsx`
- Main component. Props: `columns`, `data`, `variant`, `meta`, `actions`.
- `Column` interface: `key`, `title`, `type`, `width`, `actions`, `render` (custom override).
- `ColumnType` union: `'text' | 'date' | 'currency' | 'badge' | 'actions'`.
- `variantStyles` map for `default` vs `ghost`.
- `defaultRenderers` map: each `ColumnType` has a dedicated render function.
  - `text`: handles null/undefined/empty → `'-'`
  - `date`: `humanDate()` using `Intl.RelativeTimeFormat`
  - `currency`: shows converted amount + original amount in gray subtext if different currency
  - `badge`: blue pill span
  - `actions`: `ActionCell` subcomponent
- `ActionCell` renders `<a>`, `<form>` (with `confirm` support), or `<button>` (with `onClick` registry).
- `humanDate` replicated in JS (Today/Yesterday/n days ago).
- `currencySymbol` map replicated in JS.

### 2. `ui/src/entries/data-table.tsx`
- Hydration entry point.
- Queries all `[data-table-root]` divs on page.
- Reads adjacent `<script type="application/json">` for props.
- Passes `actionRegistry` (empty for now, extensible) to `<DataTable>`.
- Multiple roots on Daily page handled naturally (one query, loop hydrate).

### 3. `templates/partials/data-table.html`
- Go template partial.
- Accepts: `columns`, `data`, `variant`, `meta`.
- Renders container `<div data-table-root>` with nested JSON script.
- Includes `<noscript>` fallback (placeholder message — user wants proper SSR fallback later).
- Uses `jsonSafe` helper to avoid HTML escaping mangling JSON.

### 4. `ui/vitest.config.ts`
- Vitest config with `jsdom` environment, globals enabled.
- Points to `src/test-setup.ts`.

### 5. `ui/src/test-setup.ts`
- Imports `@testing-library/jest-dom` matchers.

### 6. `ui/src/components/DataTable.test.tsx`
- 7 tests:
  1. renders headers and rows for default variant
  2. renders badge type correctly
  3. renders currency with meta (converted + original amounts)
  4. ghost variant has no outer card wrapper
  5. renders action buttons (link with param replacement)
  6. custom renderer override works
  7. date formatting (Today)

## Known Issues / Follow-Up Work

### 1. Server-Rendered Fallback + CLS Prevention
**Priority:** High — user explicitly wants this.
**Status:** NOT implemented. Only `<noscript>` placeholder exists.
**What to do:** Render a basic `<table>` inside the `data-table-root` div server-side. React hydration will replace it. The SSR table should match ghost/default structure enough to prevent layout shift.

### 2. No-JS Graceful Degradation
**Priority:** Medium.
**Status:** Tables won't render without JS. The `<noscript>` message is the only fallback.

### 3. `humanDate` and `currencySymbol` Duplicated
**Priority:** Low.
**Status:** These helpers exist in Go templates AND in JS. They need to stay in sync.
**Mitigation:** The logic is simple and unlikely to change. If it does, both files need updating. Consider documenting this in code comments.

### 4. Action Registry is Empty
**Priority:** Low.
**Status:** `actionRegistry` in `data-table.tsx` has no entries. `onClick` actions in column config won't do anything unless someone populates the registry.
**Mitigation:** Not a bug — feature not yet needed. When client-side actions are needed (e.g., inline edit modal), add them to the registry.

### 5. `default` Template Function is Limited
**Priority:** Low.
**Status:** Only handles `string` and `nil`. Passing `0` or `false` won't fall through.
**Mitigation:** Only used for `variant` strings currently. Expand if needed.

### 6. `Intl.RelativeTimeFormat` Browser Support
**Priority:** Low.
**Status:** Widely supported (Chrome 71+, Firefox 65+, Safari 14.1+). Internal tool assumption holds.

## Architecture Decisions Logged

- **React island for static tables**: Usually overkill, but justified here because user explicitly wants column-definition-driven rendering (ProTable/TanStack model) and future server-side sorting/filtering. Go templates cannot cleanly express this pattern.
- **Card wrapper owned by DataTable for default, not for ghost**: Daily page's outer card contains date header + total + add-link — that's page layout, not table concern. Ghost variant returns raw `<table>` without wrapper.
- **Raw domain data passed to React, not pre-formatted**: Handlers do not format dates/currencies before serialization. Component owns formatting via type-based renderers. Custom renderers can override.
- **No new dependencies for icons**: Inline SVGs used. User wants to defer icon library decision.
- **Currency logic in component**: Reads `row.convertedAmount` || `row.amount`, `row.currency`, and `meta.currency`/`meta.currencySymbol`. Same logic as previous template markup.
